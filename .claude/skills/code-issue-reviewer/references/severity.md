# 严重程度判定矩阵

每条 finding 必须按本矩阵打分，**不得感性给分**。判定标准的核心是"现网真实风险"——这条问题如果带病上线，多大概率把人 page 起来 / 影响多大范围。

---

## 五档定义

| 档位 | 中文 | 一句话定义 |
|------|------|-----------|
| `blocker` | 阻塞 | 必须在合并 / 上线前修复；带病上线 = 必爆事故 |
| `critical` | 严重 | 强烈建议上线前修；带病上线 = 显著故障概率 / 已知大客户受影响 |
| `major` | 主要 | 应在近期排期修复；带病上线 = 偶发问题 / 部分客户受影响 |
| `minor` | 次要 | 可纳入技术债清单；不影响功能正确性 |
| `info` | 提示 | 良好实践提醒；并非缺陷 |

---

## 判定要素（同时考虑五项）

打分前对每条 finding 答这 5 个问题：

1. **触发概率**：这个问题在生产中被触发的概率？（每请求都触达 vs 仅特殊路径 vs 仅故障路径）
2. **影响范围**：触发后影响的范围？（单请求 / 单用户 vs 单租户 vs 单分片 vs 全站）
3. **影响性质**：触发后造成什么？（数据丢失 / 数据错乱 vs 服务降级 vs 单次失败）
4. **可恢复性**：是否需要人工介入？（自动恢复 vs 重试可恢复 vs 必须人工介入 vs 数据不可恢复）
5. **可观测性**：故障发生后能否快速定位？（有完整日志 / 指标 vs 完全无法定位）

## 判定矩阵（速查）

### blocker — 必须修

满足以下任意一条：
- **数据丢失 / 错乱不可恢复**：金额 / 库存 / 状态机的并发覆盖、整数溢出导致计算错误、事务回滚不一致导致脏数据
- **每请求都触达的高危路径**：核心 API 的无条件解引用 / 资源泄漏（每请求泄漏一个连接）
- **DoS 路径**：用户输入直接控制 O(n²) / 内存分配上限 / 递归深度
- **核心安全失守**：硬编码生产密钥、SQL 注入位于核心查询（注：本 skill 不深审安全，发现要标 blocker 并提示走 `/security-review`）
- **连环故障种子**：缺超时的核心调用 + 高 QPS（一次依赖故障 → 全链路雪崩）
- **吞掉关键业务异常**：金额 / 订单 / 鉴权的空 catch

### critical — 强烈建议修

- **高 QPS 路径**的 N+1 查询 / 重复反序列化 / 序列化大对象
- **可触达死锁**：明确锁顺序不一致 + 路径在生产中可能并发触达
- **写操作无幂等键** + 调用方有自动 retry
- **缓存无 TTL / 无淘汰**且持续写入
- **同步调慢服务且无超时**（非核心但仍在主流程）
- **金额 / 计数整数溢出**风险（即使概率不高）
- **金额 / 时间敏感字段**时区错误
- **错误路径无日志 + 异常对象**（事故现场无证据）
- **circular retry / retry 风暴**：紧密重试无退避
- **接口字段不兼容变更** + 已有外部调用方
- **goroutine 泄漏 / 长生命周期**：每请求泄漏 1 个

### major — 建议修

- **偶发数据竞争**但未观察到数据破坏
- **次要 API 缺超时 / 缺熔断**
- **业务事件缺 INFO 日志**（影响排障，不影响功能）
- **吞次要异常**或日志只 `e.getMessage()` 不传 exception
- **关键路径硬编码** timeout / size（应注入配置）
- **off-by-one** 或边界处理在低频路径
- **大对象在短作用域反复分配**（GC 压力）
- **listener / observer 累积**但低频
- **配置项重复定义**取值不一致
- **string 拼接性能**问题（修易、收益小）

### minor — 可纳入技术债

- 测试代码 / 工具代码的稳定性问题
- 日志措辞 / 字段命名不一致
- 内部纯函数的潜在边界
- 已被宽 catch 兜底的次要解引用
- 单环境运营场景下的时区暗坑

### info — 提示

- 良好实践提醒（如 Java 推荐用 `try-with-resources` 简化已有 close）
- 与团队习惯一致但仍有改进空间的写法
- 已被基础设施统一处理但建议代码层加强（service mesh 已有超时，仍建议代码加 fallback）
- 可读性 / 可维护性建议（仅当与稳定性间接相关）

## 升降级规则

**升级**（向上调档）：
- finding 命中**多个维度**（如 NPE + 资源泄漏在同一行）→ 取最高维度档位 +1（不超过 blocker）
- finding 在**核心业务路径**（订单 / 支付 / 登录 / 鉴权）→ 默认 +1
- finding 命中**横向去重模式 ≥ 5 处** → 模式级 finding 在原档位基础上 +1

**降级**（向下调档）：
- finding 在**测试代码 / 内部工具**（用户明确要求纳入审视时） → -1
- finding 在**已废弃模块**（含 `@Deprecated` / 注释明示） → -1
- finding 已被**容器 / 框架自动管理** → -1（资源泄漏类）
- finding 已被**基础设施统一处理**（service mesh 超时 / sidecar） → 降到 info

## 反例：常见过度评级

| 看到 | 不要直接打成 |
|-----|------------|
| 测试代码 NPE | blocker |
| 单元测试中的硬编码 IP | major |
| `println` 调试输出（debug 期遗留） | critical（除非在生产 hot path） |
| 主动让 cache 永不过期（业务确实需要） | critical（业务约束 → 可接受） |
| 短命 CLI 工具的资源泄漏 | major（OS 回收即可） |

## confidence（置信度）独立于 severity

`confidence` 反映"判断是否准确"，与"风险大小"独立：

- `high`：代码证据充分、模式明确，几乎肯定是问题
- `medium`：模式像问题，但缺少调用方上下文 / 配置上下文（需结合具体业务判断）
- `low`：仅基于关键字命中或猜测，可能误报

**low 置信度**的 finding **仍可以是 blocker / critical**——只是需要人工复核确认。但若同一 finding 多处出现都低置信，整体置信度可能升高（多证据互证）。
