# Quality Gates（质量门槛）

不是所有"踩过的坑"都值得沉淀。本文件给出最低门槛，**未达标的条目应明确告知用户"此条不入库"**，避免知识库被流水账稀释。

> 抑制膨胀比塞满更重要——一个有 30 条高质量 lesson 的库远优于一个有 300 条噪声的库。

---

## 1. 入库强制项（任一未通过即不入库）

每条经验必须满足全部下列条件：

1. **可推广**：教训段写出"未来在不同代码 / 不同模块也适用的判断"。如果只能用在被修复的那一行，不入库。
2. **有证据**：至少 1 条 evidence（代码位置 + commit / PR）。空口无凭不入库。
3. **有根因**：写出真正的根因，而非"代码 bug" 或"漏改了"。如果根因尚未弄清，先标 `status:active` 并在备注里写"待补充根因"，**不要硬塞**。
4. **有标签**：至少 `severity:` 与 `status:`；其它按相关性。
5. **有摘要**：一句话能让人靠 grep 找到。
6. **不重复**：通过候选检索确认与现有条目无重复（重复则走 MERGE）。
7. **已脱敏**：不含密码、令牌、Cookie、API key、用户邮箱、内网 IP、机密业务字段。

---

## 2. 推荐项（缺失则提示但不阻断）

- **检测预防**：给出可固化为 lint / 测试断言 / 监控指标 / 评审清单 / 设计契约的具体动作（**至少 1 项**——这一项实际上是强制的，列在这里只是提醒"不要敷衍"）
- **关联条目**：找出 ≥1 条相关 lesson 并写入 Related
- **layer 标签**：明确属于哪一层
- **lang 标签**：如果是语言相关问题
- **备注**：写明"不适用场景" / "已知反例"

---

## 3. 价值三问（要不要入库）

回答下列三问，**全部"是"才入库**：

### Q1：这条经验是否对至少 2 个未来场景有用？
- 一次性配置错误 → 否
- 个人环境问题 → 否
- 拼写 / 格式错误 → 否
- 框架选型时的考量 → 是
- 通用并发模式 → 是

### Q2：这条经验是否不是已经被既有规范 / 文档 / 自动化规则完整覆盖？
- 已被 lint 自动捕获且无新增 insight → 否
- 已是团队 wiki / 现有自动化规则的标准检查项且无新补充 → 否
- 给已有规则加新维度（适用条件 / 反例） → 是
- 全新的反模式 → 是

### Q3：这条经验如果半年后被新人翻到，能否独立读懂并立即应用？
- 大量内部缩写 / 项目特定上下文 → 否（先重写再入库）
- 通用语言 + 抽象描述 → 是
- 五段必填齐全 + 至少 1 项检测预防 → 是

---

## 4. 不入库的常见情形

下列情形，**直接告诉用户"这条不值得归档"** 并解释原因；不要硬塞：

- 一次性配置错误（密码错了、文件路径敲错、端口写错）
- 已被 lint 自动捕获的格式 / 风格问题
- 单纯的拼写、format 错误
- 个人环境问题（本地 JDK 版本不对、Node 版本不一致）
- 第三方库的临时 bug（且对方已 fix；除非是"使用了未声明的隐式假设"这种 lesson）
- 内容仅限于"某行代码改了"，无可推广见解
- 已存在等价条目且无任何新增维度
- 用户描述很模糊，连症状 / 根因都讲不清楚（先把它讲清楚再说）

---

## 5. 边界情况：值得入库 vs 不值得

| 描述 | 值得？ | 原因 |
|------|--------|------|
| "我们这次配置写错了密码" | ❌ | 一次性失误 |
| "我们部署脚本没校验密码强度，导致弱口令上线" | ✅ | 可推广为"凭据生命周期校验"教训 |
| "这个函数需要传非空 list" | ❌ | 函数级契约，写在 docstring 里就够 |
| "团队里所有 List 入参都没做空 / null 校验，靠调用方约定，已两次出 bug" | ✅ | 反映防御边界设计模式 |
| "这次 redis 挂了导致服务挂了" | ❌（仅事件） | 但这次事件**派生出**的 lesson（如"缺降级"）值得 |
| "缓存层故障未做降级导致核心读路径完全失败" | ✅ | external-resilience 类教训 |

---

## 6. 入库时的"自检清单"（贴在 PR / diff 描述里）

```
[ ] 摘要 ≤ 60 字符并可 grep
[ ] 五个必填段落齐全（症状 / 根因 / 修复 / 可推广教训 / 证据）
[ ] 至少 1 条 evidence 含 file:line + commit
[ ] 至少 1 条 detection/prevention（非空话）
[ ] severity / status 标签存在
[ ] 已查 INDEX，候选检索完成
[ ] 已脱敏，不含密钥 / PII / 内网信息
[ ] 价值三问全部"是"
[ ] INDEX.md 已同步更新（计数 / 索引行 / Next ID）
```

---

## 7. "把不值得入库的发现"也变成价值

被拒入库的发现，仍然有处理路径：
- 一次性配置错误 → 提议加**启动期校验**（这反而可能是新 lesson 的方向）
- 已被 lint 覆盖 → 不入库，但提醒用户"已有 lint 规则 X，下次留意"
- 信息不全 → 把当前已知的部分留作"待补充"草稿（不进库），等根因明朗后再决定

> 不入库 ≠ 不沟通。Claude 应该向用户讲清楚为什么不入库，并把 actionable 信息给到。
