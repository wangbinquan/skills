# 基于子代理的计划-执行技能

## 概述

本技能解决一个核心问题：当代码生成任务量大到几十甚至几百个独立文件时，单一上下文窗口无法可靠地完成所有工作。随着上下文变长，模型的注意力分散，后续生成的代码质量会显著下降（即"上下文腐化"）。

解决方案是将大任务拆解为大量独立的小任务，每5个小任务交给一个全新的、干净的子代理执行。每个子代理只需要关注它负责的5个任务和相关的设计文档片段，从而保持高质量输出。

## 工作流程总览

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  规划阶段 │ ──→ │  执行阶段 │ ──→ │  反思阶段 │
│  (Plan)  │     │(Execute)│     │(Reflect)│
└─────────┘     └─────────┘     └────┬────┘
                     ↑                │
                     │    存在未完成任务  │
                     └────────────────┘
                     全部完成 → 输出报告
```

三个阶段形成闭环，直到所有任务完成为止。

---

## 第一阶段：规划

规划阶段的目标是读懂设计文档，拆解出所有原子任务，并生成一份结构化的计划文档。

### 1.1 阅读与理解设计文档

逐个阅读用户提供的设计文档，提取以下关键信息：

- **技术栈**：编程语言、框架、依赖库版本
- **项目结构**：目录规范、包结构、模块划分
- **编码规范**：命名约定、注释要求、代码风格
- **需要生成的内容**：每个模块包含哪些文件，每个文件的职责
- **模块间关系**：哪些模块依赖哪些模块，接口契约是什么

如果设计文档不够详细，在开始分解前向用户确认关键决策点。

### 1.2 任务分解

将整个代码生成工作拆解为**原子任务**。每个原子任务应满足：

| 原则 | 说明 |
|------|------|
| **单一产出** | 一个任务对应一个文件的创建或一个文件的特定修改 |
| **自包含** | 子代理仅凭任务描述 + 设计文档即可完成，不需要查看其他任务的产出 |
| **可验证** | 完成后可以通过文件存在性或内容检查来确认 |

常见的任务类型：

- 创建实体/模型类
- 创建数据访问层（DAO/Repository）
- 创建服务层接口和实现
- 创建控制器/路由
- 创建 DTO/VO 对象
- 创建配置文件（数据库配置、中间件配置、路由配置等）
- 创建工具类/常量定义
- 创建测试文件

### 1.3 依赖分析与分层

分析任务之间的依赖关系，按依赖层级分组：

- **第1层（基础层）**：无依赖的任务。如实体类、常量定义、工具类、基础配置
- **第2层**：依赖第1层产出的任务。如 DAO 层（依赖实体类）
- **第3层**：依赖第2层产出的任务。如 Service 层（依赖 DAO 层）
- **第N层**：依次类推

同一层内的任务**互相独立**，可以并行执行。不同层之间**必须按顺序执行**。

在每一层内部，按**每5个任务一组**划分批次。

### 1.4 生成计划文档

在项目根目录生成 `PLAN.md` 文件（如果用户指定了其他位置，按用户要求）。

计划文档模板：

```markdown
# 代码生成执行计划

## 基本信息
- **设计文档**: [列出所有设计文档的路径]
- **项目根目录**: [项目路径]
- **技术栈**: [语言/框架/版本]
- **生成时间**: [YYYY-MM-DD HH:MM]
- **任务总数**: [N]
- **批次总数**: [M]
- **当前状态**: 规划完成，待执行

## 执行概览

| 层级 | 说明 | 批次范围 | 任务数 | 已完成 | 状态 |
|------|------|---------|--------|--------|------|
| 第1层 | 基础层（无依赖） | 批次1 ~ 批次K | XX | 0 | 待执行 |
| 第2层 | 数据访问层 | 批次K+1 ~ 批次M | XX | 0 | 待执行 |
| ... | ... | ... | ... | ... | ... |

## 任务详情

### 第1层：基础层（无依赖）

#### 批次 1
- [ ] **[1.1]** 创建 `src/models/User.java` — 用户实体类，包含 id/name/email 等字段。参考：设计文档 §3.1
- [ ] **[1.2]** 创建 `src/models/Order.java` — 订单实体类，包含 id/userId/amount 等字段。参考：设计文档 §3.2
- [ ] **[1.3]** 创建 `src/models/Product.java` — 商品实体类。参考：设计文档 §3.3
- [ ] **[1.4]** 创建 `src/constants/ErrorCode.java` — 错误码常量定义。参考：设计文档 §6.1
- [ ] **[1.5]** 创建 `src/config/DatabaseConfig.java` — 数据库连接配置。参考：设计文档 §2.3

#### 批次 2
- [ ] **[2.1]** ...
...

### 第2层：数据访问层（依赖第1层）

#### 批次 K+1
- [ ] **[K+1.1]** 创建 `src/dao/UserDao.java` — 用户数据访问接口。依赖：任务1.1。参考：设计文档 §4.1
...

## 执行日志
（执行过程中自动追加）
```

**关键规则**：
- 使用 `- [ ]` 表示待执行，`- [x]` 表示已完成
- 每个任务有唯一编号（如 `[1.1]`），格式为 `[批次号.批次内序号]`
- 每个任务描述包含：文件路径、简要说明、设计文档引用
- 有依赖的任务要标明依赖哪个任务

计划文档生成后，向用户展示任务总数、批次总数和层级划分，请用户确认后再进入执行阶段。

### 1.5 为每个批次提取设计摘要

这是规划阶段**最关键的步骤**，直接决定了代码生成的质量。

子代理启动时没有任何先验上下文——它看不到主对话中读取的设计文档内容。如果只给子代理一个文件路径和章节编号（如"参考设计文档 §3.1"），子代理需要自己去读取、定位、理解，这不仅浪费 token，而且容易因为定位偏差导致生成错误。

正确的做法是：**在规划阶段，主代理就把设计文档中与每个任务相关的细节提取出来**，后续直接嵌入到子代理的提示词中。

对每个任务，从设计文档中提取以下信息（按任务类型选取适用项）：

| 任务类型 | 需要提取的设计细节 |
|---------|------------------|
| **实体/模型类** | 完整的字段列表（字段名、类型、约束、说明）、实体间关联关系（一对多、多对多等）、表名和索引 |
| **DTO/VO 类** | 字段列表、与实体类的映射关系、序列化要求 |
| **Repository/DAO** | 需要定义的查询方法列表（方法名、参数、返回类型、查询逻辑描述） |
| **Service 接口和实现** | 方法签名列表（方法名、参数、返回类型）、核心业务规则、事务要求、调用的 Repository 方法 |
| **Controller/Router** | API 端点列表（HTTP方法、路径、请求参数、请求体结构、响应体结构、状态码）、认证/授权要求 |
| **配置文件** | 需要包含的具体配置项和值（数据库URL、端口、连接池大小等） |
| **测试文件** | 需要测试的方法列表、测试场景（正常流程、边界情况、异常情况） |

提取的设计摘要保存在 PLAN.md 的每个任务描述中，或保存为独立的批次摘要文件（如 `batch_001_brief.md`）。格式示例：

```markdown
#### 批次 1 设计摘要

##### [1.1] User.java — 用户实体类
字段定义：
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | Long | PK, AUTO_INCREMENT | 主键 |
| username | String(100) | NOT NULL, UNIQUE | 用户名 |
| email | String(255) | NOT NULL, UNIQUE | 邮箱 |
| passwordHash | String(255) | NOT NULL | 密码哈希 |
| status | Enum(ACTIVE,INACTIVE,LOCKED) | NOT NULL, default=ACTIVE | 账户状态 |
| createTime | LocalDateTime | NOT NULL | 创建时间 |
| updateTime | LocalDateTime | NOT NULL | 更新时间 |

关联关系：
- User 1:N Address（一个用户多个地址）
- User N:N Role（多对多，通过 user_role 中间表）

注解要求：@Entity, @Table(name="users"), @Id, @GeneratedValue, @Column

##### [1.2] UserDTO.java — 用户数据传输对象
字段：id, username, email, status, createTime（不含 passwordHash）
用途：API 响应中返回用户信息时使用
```

**为什么这一步如此重要**：子代理生成代码的质量完全取决于它收到的信息。如果提示词中只写了"创建用户实体类"，子代理只能靠猜测来定义字段；如果提示词中包含了完整的字段表、关联关系和注解要求，子代理就能生成准确的代码。多花5分钟提取设计摘要，能避免后续几十分钟的返工。

## 第二阶段：执行

### 2.1 构建子代理提示词

每个子代理的提示词必须是**自包含的**——子代理启动时没有任何先验上下文，它完成任务所需的信息必须直接写在提示词中，而不是让它自己去读文件猜测。

核心原则：**设计细节直接嵌入，不要只给路径引用**。

子代理提示词模板：

```
你是一个代码生成代理，负责完成以下代码生成任务。所有必要的设计信息已包含在下方，你无需额外读取设计文档。

## 项目信息
- 项目根目录：{项目根目录绝对路径}
- 技术栈：{语言、框架、版本}

## 编码规范
{直接列出关键规范，不要只写路径}

包结构：
  entity/    — JPA 实体类
  dto/       — 数据传输对象
  repository/ — 数据访问层
  service/    — 业务接口
  service/impl/ — 业务实现
  controller/ — REST 控制器

命名约定：
  实体类：PascalCase（如 User, Product）
  DTO：PascalCase + DTO（如 UserDTO）
  Repository：PascalCase + Repository
  Service 接口：PascalCase + Service
  Service 实现：PascalCase + ServiceImpl

注解规范：
  实体类：@Entity, @Table, @Data, @Id, @GeneratedValue
  Repository：@Repository，继承 JpaRepository<Entity, Long>
  Service 实现：@Service, @Transactional
  Controller：@RestController, @RequestMapping

## 需要完成的任务

### 任务 {编号}：{文件路径}
- **说明**：{一句话说明这个文件的职责}
- **字段/方法定义**：
  {从设计文档中提取的完整字段表或方法签名列表——这是最关键的部分}
- **关联关系**：
  {与其他实体/模块的依赖关系}
- **业务规则**：
  {代码中需要实现的具体逻辑}
- **验收标准**：
  {具体的、可检查的完成条件，如"包含 findByEmail 方法"}

### 任务 {编号}：{文件路径}
...
（最多5个任务）

## 已有代码参考（可选）
{如果前面层级已生成了代码，给出1-2个典型文件的路径，让子代理读取以保持风格一致}

## 执行要求
1. 严格按照上方提供的字段定义和方法签名生成代码，不要自行添加或遗漏
2. 创建文件前先确认目标目录存在，不存在则创建
3. 如果"已有代码参考"中列出了文件，先读取该文件以保持代码风格一致
4. 如果某个设计细节不明确，按最合理的方式实现，并在代码注释中简要说明
5. 完成所有任务后，输出完成报告，列出每个任务的状态和生成的文件路径
```

**提示词构建的核心原则——"嵌入优先，路径兜底"**：

| 信息类型 | 传递方式 | 原因 |
|---------|---------|------|
| 字段定义、方法签名 | **直接嵌入**提示词 | 这是代码生成的核心输入，不能有偏差 |
| 业务规则、约束条件 | **直接嵌入**提示词 | 子代理必须准确理解才能正确实现 |
| 编码规范（核心部分） | **直接嵌入**提示词 | 命名、注解等必须一致 |
| API 端点定义 | **直接嵌入**提示词 | HTTP方法、路径、参数不能猜 |
| 已有代码风格参考 | **路径引用**（让子代理读取） | 辅助性信息，读取1个文件即可 |
| 编码规范（完整版） | **路径引用**（如果核心部分已嵌入） | 完整规范太长时，嵌入核心部分+引用全文 |

**绝对不要这样做**：
```
❌ - **说明**：创建用户实体类
❌ - **设计参考**：参考设计文档 §3.1
```

**应该这样做**：
```
✅ - **说明**：创建用户实体类，JPA 映射到 users 表
✅ - **字段定义**：
     | 字段名 | 类型 | 约束 | 说明 |
     | id | Long | PK | 主键 |
     | username | String(100) | NOT NULL, UNIQUE | 用户名 |
     | email | String(255) | NOT NULL, UNIQUE | 邮箱 |
     ...
✅ - **关联关系**：User 1:N Address, User N:N Role
✅ - **验收标准**：包含所有7个字段，@Entity注解，继承无参构造
```

### 2.2 启动子代理

使用 Agent 工具启动子代理。核心规则：

**同一层级内的批次——并行启动：**

在同一条消息中发送多个 Agent 工具调用，每个调用对应一个批次。设置 `run_in_background: true` 让它们并行运行。

```
Agent 调用 1：批次1（任务1.1~1.5）→ run_in_background: true
Agent 调用 2：批次2（任务2.1~2.5）→ run_in_background: true
Agent 调用 3：批次3（任务3.1~3.5）→ run_in_background: true
...同层级其他批次...
```

**注意并发数量限制**：建议同时运行的子代理不超过 **5个**。如果某一层级有超过5个批次，分多轮启动——先启动前5个批次，等它们完成后再启动下一组5个批次。

**不同层级之间——顺序执行：**

必须等前一层级的所有批次都完成后，再启动下一层级的批次。因为后面的层级依赖前面层级的产出。

### 2.3 处理子代理返回结果

每当一个子代理完成时（系统会自动通知），执行以下步骤：

1. **读取子代理报告**：检查它完成了哪些任务、是否有失败的任务
2. **更新计划文档**：
   - 将已完成的任务从 `- [ ]` 改为 `- [x]`
   - 更新执行概览表中的"已完成"计数
   - 在执行日志中追加记录

执行日志条目格式：
```markdown
- [YYYY-MM-DD HH:MM] 批次 N 完成：任务 X.1 ✓, X.2 ✓, X.3 ✓, X.4 ✓, X.5 ✓
```

如果子代理报告某个任务失败：
```markdown
- [YYYY-MM-DD HH:MM] 批次 N 部分完成：任务 X.1 ✓, X.2 ✓, X.3 ✗(原因：目录冲突), X.4 ✓, X.5 ✓
```

### 2.4 进度汇报

在以下时机向用户汇报进度：
- 每完成一个层级时
- 每完成10个批次时（对于大型项目）
- 遇到子代理失败时

汇报格式：
```
进度更新：已完成 45/120 个任务（37.5%），第1层全部完成，正在执行第2层...
```

## 第三阶段：反思

当当前轮次的所有子代理都返回后，进入反思阶段。

### 3.1 计划文档审核

重新读取 `PLAN.md`，逐条检查：

1. **状态统计**：统计 `- [x]` 和 `- [ ]` 的数量
2. **文件验证**：对每个标记为完成的任务，用 Glob 工具确认目标文件确实存在
3. **遗漏检测**：检查是否有文件存在但计划中未记录的情况（可能是子代理多创建了文件）

### 3.2 决策

根据审核结果做出决策：

**情况A：全部完成**
- 所有任务标记为 `[x]`
- 所有文件通过存在性验证
- → 更新计划文档状态为"全部完成"
- → 向用户输出完成报告（见下文）

**情况B：存在未完成任务**
- 部分任务仍为 `[ ]`
- → 收集所有未完成的任务
- → 重新按每5个一组分批
- → 返回执行阶段，只执行未完成的任务
- → 在计划文档的执行日志中记录重试原因

**情况C：文件缺失**
- 任务标记为完成但文件不存在
- → 将这些任务状态回退为 `[ ]`
- → 归入未完成任务，按情况B处理

### 3.3 重试策略

- 重试时只处理未完成的任务，**不重复执行已成功的任务**
- 在子代理提示词中附加上次失败的原因（如果有），帮助子代理规避同样的问题
- **最多重试 3 轮**。如果 3 轮后仍有任务未完成，停止自动重试，向用户报告详细情况并请求指导
- 每轮重试在计划文档中记录

### 3.4 完成报告

当所有任务完成时，向用户输出以下信息：

```
## 执行完成

- 任务总数：N
- 成功完成：N
- 使用批次数：M
- 使用子代理数：M
- 重试轮次：K（如无重试则为0）

### 生成的文件清单
（按模块或层级分组列出所有生成的文件路径）

### 注意事项
（如果有任何需要用户注意的事项，如子代理做出的设计决策、发现的设计文档不一致之处等）
```

## 关键原则

### 上下文隔离
每个子代理是完全独立的进程，它看不到其他子代理的工作内容，也看不到主对话的历史。因此：
- 子代理提示词中的信息必须**完整且自包含**
- 如果需要参考已生成的代码（跨层级时），要在提示词中明确指出文件路径让子代理去读取
- 不要假设子代理知道任何背景信息

### 并发控制
- 同一层级内可并行，不同层级间必须串行
- 如果两个任务需要修改**同一个文件**，它们必须在同一个批次中由同一个子代理执行，或安排在不同轮次中串行执行
- 同时运行的子代理数量不超过5个

### 任务粒度
- 每个任务对应一个文件的创建或修改，不要把多个文件合并为一个任务
- 也不要把一个文件拆成多个任务（除非该文件非常大且可以分段生成）
- 每个批次严格包含**不超过5个**任务。不足5个时按实际数量组批

### 计划文档是唯一真相来源
- 所有任务状态的变更都必须反映在 `PLAN.md` 中
- 不要依赖记忆来追踪任务状态，始终以计划文档的内容为准
- 每次进入反思阶段时，重新读取计划文档而非依赖缓存

### 设计细节必须直接嵌入子代理提示词
子代理是一个全新的、没有任何背景知识的代理。它不知道设计文档的结构，不知道"§3.1"指的是什么，也不知道主代理在规划阶段读到了什么。如果只给路径引用，子代理要么读不到正确位置，要么花费大量 token 去重新理解整个文档——这正是我们试图通过子代理隔离来避免的上下文膨胀问题。

正确做法：
- 在规划阶段（步骤 1.5），主代理从设计文档中提取每个任务需要的**具体设计细节**（字段表、方法签名、API定义、业务规则）
- 在构建子代理提示词时，将这些细节**直接写入**提示词，而不是只给一个路径
- 子代理收到的提示词应该**自包含**——不读取任何外部文件也能完成80%以上的任务
- 只有"参考已有代码风格"这类辅助性需求才使用路径引用（让子代理读取1-2个已生成的文件）

**衡量标准**：如果把子代理提示词中的所有文件路径引用都删掉，子代理是否仍然知道要生成什么字段、什么方法、什么端点？如果不知道，说明嵌入的信息不够，需要补充。
