# TDD 代码骨架生成

## 概述

在 TDD 流程最前期生成代码骨架（结构 + 注释 + TODO），为单元测试提供清晰的接口契约。**只生成结构骨架，不生成业务逻辑**（业务逻辑由 `tdd-impl-generator` 在测试通过后实现）。

**核心原则**：
- **🚨 强制 subagent 派发（最高优先级硬约束）**：本 skill 的设计阶段（阶段 1）与所有批次生成（阶段 4）**必须通过 `Task` / `Agent` 工具显式派发独立 subagent 执行**，主上下文**严禁自行 Read 全部源文档 / 自行 Write 任务清单 / 自行生成骨架代码**。即使任务"看起来很小"也不得绕过——主上下文只做调度、回写、反思，**写代码与读大文档全部交给 subagent**。判定违规的客观信号：本应派发 subagent 的步骤里出现了主上下文的 Read/Write/Edit 工具调用（除阶段 0 的 ls/小段 Read 与阶段 4.5 的回写 Edit 外）。一旦发现立即停下，改为派发 subagent 重做。
- **主上下文精简**：设计阶段（解析文档、识别类、规划批次、落盘任务清单）由独立 subagent 完成，主上下文只收批次列表
- **完整落盘**：任务清单中每个 TaskID 必须四字段齐备（实现的业务逻辑 / 实现思路 / 关键业务规则 / 方法级实现思路），不得省略或写"同上"
- **就近落盘**：任务清单文件落盘到**原始设计文档同级目录**，命名 `<关键词>-tdd-skeleton-tasks.md`
- **批次打包**：每 subagent 承担 5~10 个 TaskID（按类规模缩放），每 wave 5~8 个 batch 并行。5~10 是推荐值非硬约束
- **严格遵从设计**：类签名、字段、方法签名按设计文档/类图直出，不得自行修改
- **注释即文档**：每个类/方法的注释必须含详细实现思路与步骤，使代码可独立审核
- **全流程自动化**：设计落盘后自动推进，仅硬冲突时询问用户

**流程六阶段**：0 续作检查 → 1 设计 subagent 落盘任务清单 → 2 自动校验/推进 → 3 语言检测 → 4 批次派发并行生成 → 5 反思补发循环 + 交付。

## 阶段 0：断点续作检查（入口闸门）

**进入技能后的第一件事**就是本阶段，不直接派发设计 subagent。

**检测**：
1. 优先目录 = 用户原始设计文档（或需求文档）所在目录
2. 用 `ls`/`find` 查找匹配 `*-tdd-skeleton-tasks*.md` 的文件
3. 多个候选时取最近修改时间那份；用户显式指定路径则优先使用

**校验**（只 Read 总览段约 50 行 + 索引表段，不读全文）：
- 总览计数齐备（总数 / done / pending / blocked / 进行中）
- 索引表行数与总数一致
- 输出路径前缀匹配项目根目录
- done 任务对应文件应已存在（缺失 > 30% 视为不可复用，回退第一阶段）

校验失败则**直接回退第一阶段重新生成**，不做就地修补，并提示用户。

**分支**：
1. `pending = blocked = 0`（已收敛）→ 跳到第五阶段直接出交付报告
2. `pending > 0` → 跳过阶段 1、2，从阶段 3 起对 pending 任务按 1.6 规则重新打包派发；不重做设计 subagent
3. 仅剩 `blocked` → 询问用户是放弃还是给新方向；放弃则进入第五阶段交付

**续作摘要**（一次性告知，不再询问）：
```
[续作] 检测到已有规划文件 → [绝对路径]
  总任务数：N，done X / blocked Y / pending Z（更新于 YYYY-MM-DD HH:MM）
  续作策略：跳过设计阶段，对 pending 任务重新打包派发
  即将自动进入语言检测与批次生成
```

**硬约束**：续作时不得重新派发设计 subagent；不得重写已 done 任务（除非用户明确要求）；不得修改已有"任务详细说明"段；新增推断/阻塞追加到原文件。若用户提供与已有规划矛盾的新需求，停下询问。

## 阶段 1：派发设计 subagent 落盘任务清单

**为什么必须独立 subagent**：原始文档体量大（需求 + 设计 + 类图 + 既有代码），主上下文解析会撑爆窗口；subagent 落盘可审计文档供团队审核；窄接口返回让主上下文只做调度。

### 1.1 设计 subagent 职责

**必做**：读取所有输入文档；解析类结构（名/类型/继承/字段/方法签名/注解）；建立类清单标注来源与推断；划分批次（每 batch 5~10 TaskID）；为每个 TaskID 写完整四字段说明；用 Write **一次性**落盘；按 1.7 格式回传 YAML 批次列表。

**禁做**：生成任何代码骨架；回传任务清单全文或大段摘录；以"内容重复"为由省略任何 TaskID 的四字段；修改用户原始文档。

### 1.2 落盘位置与命名

- **目录**：与**原始设计文档同级**（优先）；只有需求文档则用其目录；都没有具体路径则在派发前询问用户
- **命名**：`<关键词>-tdd-skeleton-tasks.md`，关键词取自设计文档标题或用户描述中的核心主体
- **冲突处理**：同名已存在 → 追加 `-YYYYMMDD-HHMM`，**禁止覆盖**

### 1.3 设计 subagent prompt 模板

> **🚨 派发动作硬约束**：本节的"派发"必须是**真实的 `Task` / `Agent` 工具调用**——即在主上下文的本轮回复中出现一个 `Agent` 工具块，subagent_type=`general-purpose`，并把下方 prompt 整体作为 `prompt` 参数传入。**严禁**主上下文自己 Read 原始文档、自己规划类清单、自己 Write 任务清单文件来"代替"这次派发。如果主上下文准备直接动手做这些，立即停下并改为派发 subagent。

主上下文用 `Agent` 工具（subagent_type=`general-purpose`）派发 1 个设计 subagent，prompt 必须自包含：

````
任务：作为 TDD 代码骨架的"设计 subagent"，独立完成文档解析、类结构识别、批次规划，并将完整任务清单文件落盘。

【原始输入】
- 需求文档：[路径或内容]
- 设计文档：[路径或内容]
- 类图/UML（独立文件）：[路径或内容]
- 既有代码：[路径]

【项目语言（若已知）】[语言名 或 "待检测"]

【任务清单文件落盘绝对路径】[由主上下文按 1.2 规则给出]

【落盘文件结构要求】必须完整包含：
1. 总览（项目、语言、参考文件、原始文档路径、各状态计数、当前批次、补发轮次）
2. 任务明细索引表（TaskID、类名、类型、来源、依赖类型、输出路径、批次、状态、轮次、备注）
3. 任务详细说明区（每 TaskID 一节，四字段齐备：实现的业务逻辑、实现思路、关键业务规则/设计约束、方法级实现思路）
4. 批次计划表（批次号、并行任务数、TaskID 列表、备注）
5. 依赖类型字典（类型名、文件路径、种类）
6/7/8. 推断项汇总 / 阻塞项汇总 / 批次执行日志（初始留空占位）

完整模板见附件 1.4。

【类图解析规则】见附件 1.5。
【批次划分原则】见附件 1.6。

【硬性要求】
1. Read 工具读取所有输入文档
2. Write 工具一次性落盘完整任务清单
3. 不得省略任何 TaskID 的四字段；即使内容长也必须全部写出
4. 四字段必须从需求/设计文档直接映射；无明确出处时标注 `[推断：<具体说明>]`
5. 不要把任务清单全文粘到返回消息中
6. 不要生成任何代码骨架

【返回格式】仅返回如下 YAML（可附一句话确认）：

```yaml
design_output:
  tasks_file: [落盘绝对路径]
  language: [检测或沿用]
  total_tasks: N
  batches:
    - batch_id: batch-1
      wave: 1
      task_ids: [TASK-001, TASK-002, ...]   # 5~10 个
      size_class: small | medium | large
      summary: [一句话业务领域]
    - batch_id: batch-2
      wave: 1
      task_ids: [...]
      size_class: medium
      summary: ...
  unresolved_questions:
    - "[仅填写阻塞性问题：文档矛盾、关键信息缺失、推断风险高]"
  inferred_items_count: N
  blocked_tasks: []
  notes: "[可选，一两句]"
```

batch = 一个 subagent 工作单元（5~10 TaskID）；wave = 一波并行调度（5~8 batch）。有依赖的 batch 须分到不同 wave。
````

### 1.5 类图解析规则

```
优先级：
1. 类图明确定义 → 直接使用，不可修改
2. 类图存在但类型不明 → 上下文推断，注释说明依据
3. 类图未覆盖但需求需要 → 注释 `[推断：待用户确认]`
```

**严禁**：添加类图未定义的方法；改动访问修饰符；改动方法签名（参数顺序/类型/返回值）。

### 1.6 批次划分原则

| 类规模 | 每 batch 推荐装多少 | 判定 |
|--------|---------------------|------|
| small（方法 ≤ 8） | 8~10 | 字段简单、注释量小 |
| medium（方法 9~15） | 6~8 | 常见业务服务 |
| large（方法 16~25） | 3~5 | 结构复杂、依赖多 |
| extra large（方法 > 25） | 1~2 | 接近 subagent 单任务上限；> 40 必要时方法组拆分 |

**应避免**（非硬性违规）：单 batch 仅 1 个普通类；单 batch > 10 类或 > 80 方法；跨业务域混装。

**wave 组织**：无依赖优先 wave 1；同一业务域尽量同 batch；wave 内工作量均衡；每 wave 5~8 batch（超出拆多 wave）。

> 5~10 是推荐值，主上下文校验时不会因偏离此范围而强制重划分。

### 1.7 设计 subagent 返回收窄

**只能**返回：一句话确认 + `design_output` YAML。**禁止**返回：任务清单正文、四字段内容、解析过程。

## 阶段 2：自动校验与推进（不询问用户）

### 2.1 解析与校验

主上下文收到 design_output 后：
1. 提取 `tasks_file`、`total_tasks`、`batches`、`unresolved_questions`、`inferred_items_count`
2. 按需 Read：总览段（确认 total_tasks 与索引表行数一致）+ 1~2 个代表性 TaskID（确认四字段齐备）+ 批次计划表
3. 字段齐备且无硬约束违规 → 输出一行摘要直接进入阶段 3
4. 字段省略或硬约束违规 → 跳到 2.2 自动修订

> batch 规模偏离 5~10 **不触发**修订（5~10 是推荐值）。

**摘要**：
```
[设计完成] 任务清单已落盘 → [绝对路径]
  总任务数：N，打包为 M 个 batch，分 K 波；推断项 P 条
  自动进入语言检测与批次生成
```

### 2.2 自动修订

字段缺失或计数错乱时主上下文自行派发"设计修订 subagent"（不询问），prompt 列出问题、要求 Edit 就地修复、保持字段齐备、更新计数；返回更新后的 `design_output`。回到 2.1 重校；连续 2 轮失败则进入 2.3。

### 2.3 硬阻塞（仅此时询问用户）

仅在以下情况暂停：
1. `unresolved_questions` 含阻塞性问题（文档矛盾、关键类型缺失等）
2. 自动修订连续 2 轮仍不达标
3. 推断项 > 30% 或推断涉及核心业务逻辑
4. 输出路径与既有代码冲突

询问格式：
```
[需要决策] 检测到 N 个阻塞问题：
1. [问题] — 默认处置：[xxx]，接受请回复 "go"，否则说明调整
```

## 阶段 3：语言检测与加载参考资料

### 3.1 语言检测

1. 优先用 `design_output.language`；若"待检测"继续：
2. 看用户提供文件扩展名
3. 查项目根目录标志文件：
   - Java：`pom.xml`/`build.gradle`/`*.java`
   - C++：`CMakeLists.txt`/`Makefile`/`*.cpp`/`*.hpp`/`*.h`
   - Python：`requirements.txt`/`setup.py`/`pyproject.toml`/`*.py`
   - Go：`go.mod`/`go.sum`/`*.go`
   - Rust：`Cargo.toml`/`Cargo.lock`/`*.rs`
   - Scala：`build.sbt`/`*.scala`
   - JavaScript：`package.json`（无 `tsconfig.json`，源码 `*.js`/`*.mjs`/`*.cjs`）
   - TypeScript：`tsconfig.json`/`*.ts`/`*.tsx`；JS+TS 共存按 TS
4. 仍无法确定 → 询问用户

### 3.2 加载参考文件（只加载对应语言，避免上下文过载）

参考文件位于 `references/<语言>.md`（java、cpp、python、go、rust、scala、javascript、typescript）。各文件包含该语言的注释规范、类/接口/抽象类/枚举/trait/struct 模板、TODO 规范、空方法体占位约定、命名约定。

主上下文用 Edit 工具更新任务清单文件总览区"参考文件"字段为绝对路径（仅 Edit 单行，不加载全文）。

## 阶段 4：批次派发并行生成

### 4.1 编写规范（所有生成 subagent 必须遵守）

**类级注释**（语言标准格式，必须齐备）：职责描述 / 设计思路（架构位置、设计模式）/ 实现思路（核心技术方案、数据结构、线程模型）/ 主要依赖 / 线程安全性 / 设计约束 / 来源标注。

**方法级注释**（必须齐备）：方法职责 / 实现思路 / **实现步骤**（分步骤列出，是 TODO 的依据）/ 参数说明（含约束）/ 返回值说明（含 null/empty/异常）/ 异常说明 / 前置后置条件 / 并发注意。

**TODO 注释**：精确描述待实现内容，分步骤、具体到调用、抛哪些异常。
- 正确：`// TODO: Step 1 - 校验 userId 不为空且 UUID 格式；越界抛 IllegalArgumentException`
- 禁止：`// TODO: implement this method`

**空方法体规范**（合法编译占位）：
- Java/Scala：返回 `null` 或 `0/false`，附 `// TODO: implement`
- Python：`raise NotImplementedError("TODO: implement <name>")`
- Go：返回零值与 `fmt.Errorf("not implemented")`
- Rust：`unimplemented!("TODO: implement <name>")`
- C++：纯虚不需函数体；非纯虚在 cpp 放 TODO 注释并返回零值

详细模板见 `references/<语言>.md`。

### 4.2 调度规则

> **🚨 派发动作硬约束（再次强调）**：本阶段每一个 batch **必须以一个真实的 `Task` / `Agent` 工具调用**形式发出，而**不是**主上下文自己 Read 骨架文件 + 自己 Write 代码骨架。同 wave 多 batch = 同一条消息里多个 `Agent` 工具调用块并排出现。**违规判定**：本阶段中主上下文出现 `Write` 骨架文件、或 `Read` 任意 TaskID 输出路径、或自己生成代码内容——立即停下，回退到本节重新以 subagent 形式派发。**例外**：4.3 的"按需 Read 任务清单 TaskID 段"与 4.5 的"回写任务清单"是允许的主上下文动作。

1. **按 wave 派发**：Read 任务清单"批次计划"表（只读这一段），一次派发整 wave 的全部 batch（5~8 并行）
2. **同消息内并行**：同 wave 所有 subagent 在同一条消息中发起多个 `Agent` 工具调用块（注意：**必须真实调用 `Agent` 工具**，而非主上下文文字描述"派发了 N 个 subagent"）
3. **每 subagent 自包含**：subagent 看不到会话历史，prompt 必须自带全部信息
4. **一个 subagent = 一个 batch = 5~10 TaskID**（按类规模缩放），顺序处理这些类
5. **wave 屏障**：本 wave 全部返回前不派发下一 wave；返回后立即回写

### 4.3 按需读取本 batch 的详细说明

派发前主上下文从任务清单文件 Read 出**本 batch 全部 TaskID** 的详细说明段（每段从 `### TASK-XXX:` 到下一个 `### TASK-` 之前），按顺序粘到 prompt 的"任务详细说明"区。

**允许**：一次 Read 本 batch 的 5~10 个 TaskID 段（约 20~60 KB）。
**禁止**：把整份任务清单载入主上下文；粘贴非本 batch 的 TaskID；跨 batch 混合。
**操作**：用 Read 的 `offset`/`limit` 定位行范围。

### 4.4 生成 subagent prompt 模板

````
批次 ID：batch-N（wave-K）
任务：为以下 5~10 个类生成代码骨架，**按清单顺序逐个处理**。本 subagent 只处理列出的 TaskID。

【本批次 TaskID 清单】
- TASK-XXX: [类名]（[类型]）→ 输出：[路径]
[...]

执行策略：按清单顺序，每个类独立完成"读需求 → 生成代码 → Write 到输出路径 → 进入下一个"。每个类独立写文件，不得合并。

【每个 TaskID 的详细说明】（主上下文从任务清单 Read 后按顺序粘贴）

#### TASK-XXX: [类名]
**目标类设计**：[来源、类型、字段列表、方法签名、继承关系、注解 — 完整粘贴]
**相关业务需求**：[与本类相关的需求段落]
**实现的业务逻辑** / **实现思路** / **关键业务规则** / **方法级实现思路**：
[全文粘贴，不省略]
**输出路径**：[绝对或相对]

[...每个 TaskID 都完整粘贴，不得"同上"]

【依赖类型字典】（这些类型由其他 subagent 生成，按签名引用即可）
- [类型] → kind=interface/class → [路径]

【项目语言与规范】
语言：[语言]　参考文件（必读一次）：[绝对路径 references/<语言>.md]
严格遵循其中的类级注释、方法级注释、TODO 注释、空方法体占位规范。

【硬性要求】
1. 严格按设计文档的类名、方法签名、字段定义生成，不得自改
2. 类级注释必须齐备且用中文（除代码元素名）
3. 每个方法（含 getter/setter）必须有方法级注释
4. 方法体只放 TODO，TODO 须精确描述"调哪个依赖、调哪个方法、处理哪些条件、抛哪些异常"
5. 推断未明确内容时注释标注 `[推断：<说明>]`，并在回执对应 TaskID 下记录
6. 空方法体使用合法编译占位
7. 每个 TaskID 一个 Write（不要合并文件、不要 Edit 合并）
8. 处理完一个 TaskID 再下一个

【进度回执（必须，YAML，每个 TaskID 一条）】
```yaml
batch_progress:
  batch_id: batch-N
  items:
    - task_id: TASK-XXX
      class_name: [类名]
      status: done   # done / blocked / pending
      output_file: [实际写入绝对路径]
      methods_generated:
        - name: [方法名]
          signature: "[完整签名]"
      inferred_items:

          name: [名称]
          reason: "[依据]"

        - detail: "[偏离]"
          reason: "[原因]"
      notes: "..."
    [...本 batch 每个 TaskID 都必须有一条，即使 blocked/pending]
```

某 TaskID 无法完成时，将其 status 设为 blocked 并详细说明 reason，**继续处理其他 TaskID**，不得因一个失败就放弃整批。
````

### 4.5 每 wave 完成后的回写（主上下文单线写入）

本 wave 全部 subagent 返回后，主上下文**立即**：

1. 解析每个 batch 的 `batch_progress.items`，提取每条的 task_id/status/output_file/methods_generated/inferred_items/deviations
2. 用 Edit 就地更新任务清单（定位单行，不 Read 全文）：
   - 索引表"状态"列按 status 改 `done`/`blocked`；未在任何 items 中出现的 TaskID 置为 `pending`（隐式漏报）
   - 总览区 done/pending/blocked 计数
   - 推断项汇总区追加所有 inferred_items
   - 阻塞项汇总区追加所有 blocked 任务及 reason
   - 批次执行日志区追加本 wave 情况
3. 更新依赖类型字典：新 done 的类型加入字典供后续 wave 引用

**为什么主上下文写而非 subagent**：避免并发 Edit 冲突；唯一仲裁点保证反思读取可信状态；窄接口（结构化 YAML）比直接改 markdown 稳定。

### 4.6 派发下一 wave

仍有未开始的 wave → 回到 4.2。全部派发完 → 进入阶段 5。

## 阶段 5：反思补发循环 + 交付

强制循环：读总览 → 完整性反思 → 补发漏生成 → 回写 → 再反思，直到 `pending = 0`。

### 5.1 完整性反思

只 Read 总览区与索引表（不 Read 详细说明区），统计：

```
[反思] 任务清单共 N 项：done X，blocked Y，pending Z
       pending 归属：TASK-005、TASK-012
       blocked 归属：TASK-009 (reason=...)
       判定：[未收敛 / 已收敛]
```

**收敛**：`pending = 0`（只剩 done 和已解释的 blocked）。**未收敛**：存在 pending 必须进入 5.2。

**兜底**：subagent 漏报的任务在 4.5 已置为 pending，原因"subagent 未报告"。

### 5.2 补发循环（强制执行）

对所有 pending：
1. 按相同打包规则重新打包（5~10 TaskID/subagent，pending ≤ 4 时不强求凑满，按实际数打成一个 batch）
2. 派发补发 wave（5~8 batch 并行）
3. 补发 prompt 必须：
   - 标注"补发任务（轮次 = 当前 + 1）"
   - 用 4.4 多 TaskID 模板列出全部 TaskID
   - 附上依赖类型字典最新版（可能有新 done 的类型可引用）
   - 附上前一轮失败原因（按 TaskID 分别列）
   - 主上下文继续按需 Read 各 TaskID 详细说明并按顺序粘贴
4. 返回后回到 4.5 回写 → 5.1 反思
5. 循环直到无 pending

**硬性**：不可跳过补发；同一 TaskID 连续 2 轮 pending 且原因不同则暂停询问用户（防死循环）。

### 5.3 一致性检查（收敛后一次）

- 命名风格符合目标语言规范
- 包/模块路径符合预期
- 注释统一中文（代码元素名除外）
- 依赖注入方式统一
- import/using 无冗余
- 抽样 3 个文件复检注释字段齐备（缺失则派发补齐 subagent）

### 5.4 推断项汇总（不阻塞交付）

从"推断项汇总"区收集所有 `[推断：...]` 项按类分组到交付报告"推断项清单"。**不请求用户逐条确认**，用户可在交付后按需触发调整。

仅以下情况主动暂停（与 2.3 硬阻塞一致）：推断项 > 30%；某推断涉及核心业务逻辑（鉴权/金额/状态机）；多推断相互依赖致架构偏离。

### 5.5 交付报告

```
## 代码骨架生成交付报告

### 收敛情况
- 任务总数：N　done X　blocked Y　pending 0（必须 0）
- 补发轮次：K

### 生成概览
| TaskID | 类名 | 类型 | 文件路径 | 方法数 | 推断项数 |

### 推断项清单（仅供审阅，无需逐条确认）
[按类分组列出推断项]
> 如需调整，指定 TaskID + 调整方向，将派发修订 subagent。

### Blocked 项 & 建议
- [TaskID/类名]：[reason] → [建议：集成测试/手工实现/澄清设计]

### 任务清单文件
- 路径：[落盘绝对路径]

### 下一步（TDD 流程衔接）
1. 审核推断项清单，需调整可重新生成对应类
2. 审核每方法的"实现步骤"注释
```

## 附：小规模简化流程

| 类数量 | 简化策略 |
|--------|---------|
| 1~2 | 主上下文直接生成，会话内维护任务清单；仍需推断项标注、注释齐备、TODO 精确、自动化 |

| > 10 | 必须走完整五阶段：设计 subagent 独立派发、任务清单落盘、5~10 TaskID/subagent、5~8 batch/wave、每 wave 回写、反思循环至 pending = 0 |

全规模均"全程自动化"——仅硬阻塞下询问用户。
